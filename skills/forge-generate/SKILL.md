# Forge Generate

Read the plan. Read the L0 templates. Read the survey. Produce L2 skills that know this repo. Every L2 skill is fully self-contained — no references back to L0.

<HARD-GATE>
Do not generate without a completed plan. If `.claude/forge/plan.md`
doesn't exist, stop and run forge-plan first.
</HARD-GATE>

<HARD-GATE>
Every reference in a generated L2 skill must point to something REAL in
the target repo — real file paths, real commands, real test helpers, real
patterns. A skill that references a nonexistent helper is worse than
a generic skill that doesn't reference anything.
</HARD-GATE>

<HARD-GATE>
L2 skills are fully self-contained. They do NOT import, reference, or
depend on L0 templates at runtime. When L0 improves, re-running the
forge regenerates L2. The forge IS the update mechanism.
</HARD-GATE>

## Input

Target repo path (same repo with `survey.md` and `plan.md` in `.claude/forge/`).

The agent also needs access to the L0 auto-* templates. These live in the ai-workflow-config repo at `skills/auto-*/SKILL.md`. The agent reads them as raw material — the structure, hard gates, anti-patterns, and process steps that get customized with repo-specific knowledge.

## Process

### 0. Load Everything

1. Read `.claude/forge/plan.md` — what to generate
2. Read `.claude/forge/survey.md` — repo-specific data
3. Read `context/` in the target repo (if exists) — architectural truth
4. Read the L0 templates listed in the plan's "Base L0" fields

### 1. Foundation Work (if planned)

If the plan calls for foundation work, do it first — downstream skills may depend on infrastructure being in place.

#### CLAUDE.md

If missing or incomplete, generate from survey data:

```markdown
# [Project Name]

[One-paragraph description from survey summary]

## Build & Test

```bash
[verified build command from survey]
[verified test command from survey]
[verified lint command from survey]
[verified format command from survey]
```

## Project Structure

[From survey, annotated with where new code goes]

## Conventions

[From survey code conventions — error handling, naming, module organization]

## Key Dependencies

[Top 5-10 from survey]
```

Verify the CLAUDE.md by running every command it lists.

#### context/ Bootstrap

If `context/` doesn't exist and the survey found enough architectural information, create minimal context files:
- `context/architecture.md` — project structure, module boundaries, key patterns
- `context/testing-strategy.md` — test infrastructure, mock boundaries, fixture patterns

Mark all content as "Generated by forge — not yet human-confirmed."

#### CI/CD

If the plan flags missing CI, generate a foundation skill that knows how to set it up. The actual CI creation happens when the foundation skill runs, not during forge generation.

### 2. Generate Domain Skills

Domain skills before process skills — process skills may reference domain knowledge.

For each domain skill in the plan:

1. **Read the plan entry** — purpose, key knowledge to capture
2. **Read relevant survey sections** — code snippets, patterns, conventions
3. **Read relevant source files** (if the plan says "forge-generate reads directly")
4. **Write the skill** in `.claude/skills/[name]/SKILL.md`

Domain skills have no L0 base — they're written from scratch based on survey + codebase patterns.

#### Domain Skill Structure

```markdown
---
name: [name]
description: [one-line: what knowledge this captures]
when_to_use: [when an agent should load this skill]
allowed-tools: [Read, Grep, Glob — domain skills are reference, not action]
---

# [Skill Name]

[What this skill teaches an agent about this specific codebase]

## [Topic Sections]

[Repo-specific knowledge organized by topic. Code snippets showing
the RIGHT way to do things in this codebase. Anti-patterns specific
to this project.]
```

### 3. Generate Process Skills

For each process skill in the plan:

1. **Read the corresponding L0 template** (e.g., `auto-plan-grill/SKILL.md`)
2. **Read the plan entry** — what customizations to make
3. **Produce the L2 skill** by adapting L0 with repo-specific knowledge

#### The Adaptation Process

The L0 template provides the STRUCTURE — phases, hard gates, anti-patterns, process flow. The survey provides the CONTENT — repo-specific commands, patterns, conventions.

For each section of the L0 template:

| L0 Section | Adaptation |
|---|---|
| **Frontmatter** | Update description with project name and specifics |
| **Hard gates** | Keep all L0 hard gates. Add project-specific gates if warranted. |
| **Input/output format** | Customize file paths, conventions from survey |
| **Process steps** | Replace generic guidance with specific commands, patterns, file locations |
| **Anti-patterns** | Keep L0 anti-patterns. Add project-specific anti-patterns from survey. |
| **Verification** | Replace generic checks with specific commands from survey |

What to INJECT (repo-specific):
- Exact build/test/lint commands (verified by survey)
- Test framework and runner specifics
- File naming and organization conventions
- Error handling patterns with code examples
- Mock boundary decisions with rationale
- Project-specific constraint categories for grill lenses
- Where new code goes (module, crate, directory)

What to KEEP from L0:
- All hard gates (locked tests, read-before-write, no implementation in test skills)
- Process flow (the phases don't change)
- Anti-pattern tables (add to, don't replace)
- GitHub communication patterns
- Confidence model

What to REMOVE:
- Generic "find the test framework" steps (replaced with specific framework)
- "Discover conventions" steps (replaced with known conventions)
- Hedging language ("typically," "usually") replaced with definitive statements

#### L2 Process Skill Structure

Same structure as L0, but every generic placeholder is filled with repo-specific values:

```markdown
---
name: [name]
description: >-
  [L0 description, adapted: "for [project-name]" + specific details]
when_to_use: >-
  [L0 when_to_use, adapted for this project's workflow]
allowed-tools: [same as L0]
argument-hint: "[project-specific hint]"
effort: high
---

# [Skill Name] — [project-name]

[L0 intro, adapted with project context]

[HARD GATES from L0 — kept verbatim or strengthened]

## [L0 sections with repo-specific content injected]
```

### 4. Generate Foundation Skill (if planned)

The foundation skill handles infrastructure and initialization. It's conditional — only generated if the survey found gaps.

```markdown
---
name: foundation
description: >-
  Project initialization and infrastructure for [project-name].
  Handles CI/CD setup, missing infrastructure, and structural additions.
when_to_use: >-
  Use when the project needs new infrastructure before feature work
  can begin. Triggered by forge when gaps are found, or
  manually when adding major structural components.
allowed-tools: [Read, Write, Edit, Bash, Task]
---

# Foundation — [project-name]

## What This Handles
[From plan's foundation work section]

## CI/CD Setup
[Specific to this project's language, build system, and hosting]

## Success Criteria
- Build passes: `[command]`
- Tests pass: `[command]`
- Lint passes: `[command]`
- CI pipeline runs on PR
```

### 5. Generate gyre.yaml

From the plan's gyre config section:

```yaml
# .claude/gyre.yaml
name: [project]-agent
description: Autonomous dev agent for [project-name]

skills:
  - skill: plan-grill
    triggers: [issue-created, manual]
    outputs: [plan, open-questions, spec-stubs]

  - skill: test-planning
    triggers: [plan-complete, slice-unblocked]
    inputs: [plan, spec-stubs]
    outputs: [test-contracts]
    condition: "slice has no test contract yet"

  - skill: test-writer
    triggers: [contract-validated]
    inputs: [test-contracts]
    outputs: [locked-tests]
    condition: "slice has validated contract but no committed test"

  - skill: build
    triggers: [tests-committed]
    inputs: [locked-tests, spec]
    outputs: [implementation]
    condition: "slice has committed red test"

  - skill: refactor
    triggers: [slice-built]
    inputs: [implementation-diff]
    outputs: [refactored-code]

  [- skill: foundation  # only if generated]

github:
  branch_prefix: "agent/"
  commit_style: "[from survey git conventions]"
  pr_labels: [agent-in-progress, needs-answer, ready-for-review, agent-slice, agent-plan]

verification:
  build: "[from survey]"
  test: "[from survey]"
  lint: "[from survey]"
  format_check: "[from survey]"
```

### 6. Validation Pass

Before committing, validate every generated artifact:

For each L2 skill:
- [ ] Every command referenced exists and was verified in the survey
- [ ] Every file path referenced exists in the target repo
- [ ] Every test helper/pattern referenced appears in actual test files
- [ ] No generic placeholders remain (`[project-name]`, `[command]`, etc.)
- [ ] Hard gates from L0 are preserved
- [ ] Frontmatter is valid (name, description, when_to_use, allowed-tools)

For CLAUDE.md:
- [ ] Every command listed actually works (re-run if unsure)
- [ ] Project structure matches reality
- [ ] Conventions match what the survey found

For gyre.yaml:
- [ ] All referenced skills exist in `.claude/skills/`
- [ ] Commands match survey
- [ ] Commit style matches survey git conventions

### 7. Commit and Push

1. Commit all generated files:
   ```
   feat: generate L2 gyre from forge
   ```
2. Push to branch
3. Open or update PR with generation summary:
   ```
   📋 STATUS: Gyre generated
      Process skills: [list]
      Domain skills: [list]
      Foundation: [yes/no]
      CLAUDE.md: [generated/updated/unchanged]
      gyre.yaml: generated
      Next: forge-verify for smoke testing
   ```

## Output

This skill produces (in the target repo):
1. `.claude/skills/*/SKILL.md` — L2 process and domain skills
2. `.claude/gyre.yaml` — gyre manifest
3. `CLAUDE.md` — project-level docs (if missing/incomplete)
4. `context/` bootstrap files (if context/ didn't exist)
5. `.claude/forge/plan.md` updated with generation status

## Anti-Patterns

| Anti-Pattern | Fix |
|---|---|
| **Referencing L0 templates from L2 skills** | L2 is self-contained. Copy and adapt, don't reference. |
| **Leaving generic placeholders** | Every `[placeholder]` must be filled with real values. |
| **Referencing nonexistent files** | Validate every path. A broken reference is worse than none. |
| **Removing L0 hard gates** | Keep all hard gates. They exist for reasons. |
| **Over-generating domain skills** | Only generate domain skills the plan calls for. Not every project needs a security skill. |
| **Generating without verifying commands** | The survey verified them. If the survey didn't, re-verify now. |
| **Making L2 skills too long** | L2 should be SHORTER than L0 — specific guidance replaces generic exploration. |

## Handoff

After this skill completes:
- **forge-verify** runs smoke tests on the generated gyre
- The human reviews the generated skills via PR
- After verification and review, the gyre is ready for autonomous use
